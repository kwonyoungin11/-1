# 개발 루프 (DEV LOOP) — 오너 확정

**확정일:** 2026-07-09  
**오너 선택:** 옵션 1 — **개발 루프로만** 공식 채택  
**출처 개념:** AI “loop” (목표 → 실행 → 검증 → 수정 → 반복).  
**이 프로젝트 적용 범위:** 소프트웨어 개발·검증만. **실주문 자동 루프 아님.**

---

## 쉬운 설명

| 기술 말 | 쉬운 말 |
|---------|---------|
| 개발 루프 | 목표를 정하고, 만들고, **검사**하고, 실패하면 고쳐 다시 검사. 통과할 때까지(또는 한도까지) 반복 |
| Verify | 사람 느낌이 아니라 **테스트·스캔**이 “합격/불합격”을 말함 |
| Stop condition | 무한 반복 금지. 성공하거나, 횟수 한도에 걸리면 멈춤 |
| Maker ≠ Checker | 만드는 일과 검사하는 일을 섞지 않음 |

**비유:** 작업실(worktree)에서 제품을 만들고, 검사 기계가 통과할 때만 정식 매장(main)으로 보냄.

---

## 왜 필요한가? (실거래 목적과 연결)

- 최종 목적은 **실거래**다.
- 실거래를 열려면 중간에 쌓는 코드·화면·연결이 **매번 검증**되어야 한다.
- 루프는 “빨리 대충”이 아니라 **검증 가능한 속도로** 가기 위한 방법이다.

## 돈을 잃을 위험이 있는가?

| 루프 종류 | 허용 | 이유 |
|-----------|------|------|
| 개발 루프 (test/scan) | **허용** | 주문 API를 돌리지 않음 |
| paper / dry-run 반복 (향후) | 설계 후 | 실주문 없음, 별도 승인 |
| **실주문 자동 루프** | **금지** (Phase 6 전) | 돈 손실·중복 주문 위험 |

---

## 공식 사이클 (에이전트·사람 공통)

```text
1. 목표 한 줄 고정 (이번 작업이 실거래 로드맵의 어디에 붙는지)
2. worktree에서만 작업 (main 직접 수정 금지)
3. 구현 / 문서 수정
4. 검증 (Checker):
   - bash scripts/grok/dev-loop.sh
   - 또는: secret scan → trading safety scan → dotnet test
5. 실패하면 원인 수정 후 4로 돌아감
6. 성공 조건 충족 시:
   - 커밋
   - (필요 시) main 병합 + GitHub push
   - 오너에게 쉬운 말로 보고
7. 한도 초과 시:
   - 즉시 중단
   - 실패 요약 + 오너 결정 요청
   - 실주문 관련 플래그 절대 완화 금지
```

---

## 성공 / 실패 판정 (객관적)

### 기본 게이트 (매 개발 루프)

1. `scripts/grok/check-secrets.sh` PASS  
2. `scripts/grok/check-trading-safety.sh` PASS  
3. `dotnet build` / `dotnet test` PASS (SDK 있을 때)  
4. live 기본 차단 유지 (`ALLOW_LIVE_ORDERS=false` 등)

### 금지 (루프 안에서도)

- `.env` 내용 출력·커밋  
- Toss 주문 create/modify/cancel 호출  
- kill switch / live gate 우회  
- 검증 실패를 무시하고 병합  

---

## 종료 조건 (Stop)

| 조건 | 동작 |
|------|------|
| 모든 기본 게이트 PASS | 루프 성공 → 커밋/보고 |
| 같은 목표로 **최대 5회** 검증 재시도 후에도 FAIL | 중단 → 오너 보고 |
| secret / live-safety **BLOCK** | 즉시 중단 (재시도로 우회 금지) |
| 목표 자체가 모호 | 구현 전에 오너 결정 요청 |

`dev-loop.sh` 기본: `MAX_ATTEMPTS=5` (환경변수로 조정 가능, 상한 권장 10).

## Maker / Checker 분리

| 역할 | 하는 일 | 하지 않는 일 |
|------|---------|----------------|
| Maker (구현) | 코드·문서·플랜 작성 | “이 정도면 됐겠지”로 스캔 생략 |
| Checker (검증) | test + safety scripts | 기능 설계를 대충 통과시킴 |

한 에이전트가 둘 다 하더라도 **순서와 게이트는 분리**한다.  
검사 명령이 빨간불이면 병합하지 않는다.
